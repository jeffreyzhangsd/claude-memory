For every pipeline that uses a local Ollama model in `jef-local-agents`, run an A/B comparison against the realistic alternatives on real input before locking the model choice. Build the pipeline with a `--model` flag so swapping is trivial. Report quality + latency. Pick the winner on data, not vibes.

**Why:** Hermes 8B is agentic-tuned but smaller than qwen3:14b and only marginally larger than gemma4:e4b. Initial assumption "Hermes is the agent model so it wins for any reasoning task" turned out to be wrong — Hermes wins for some tasks (graded decisions) and loses for others (prose generation). User flagged early that adding Hermes without testing is "clutter," and we want to keep the local stack lean. See [[agentic-batching]] for the constraint that only one model is hot at a time.

**Results so far (keep updating as new pipelines ship):**

- **Job grading (`career/grade.py`)** — hermes3:8b wins. vs gemma4:e4b: verdict parity, gemma ~1.7x slower. vs qwen3:14b: over-decisive on borderline cases (demoted a real "maybe" to "no, 9"). Hermes is correctly cautious on borderline.
- **Cover letter pre-fill (`career/prefill.py`)** — gemma4:e4b wins decisively. Hermes paraphrases the CV's Professional Summary instead of leading with concrete metrics, uses bullet lists inside the cover letter (against prompt rules), and fails the "verbatim CV bullet" rule on tailoring memo. Gemma also ~4-7s faster. Verified 2026-05-14 on Spotify Full-Stack and Twitch SWE I JDs.
- **Wiki orderer (`wiki_builder.make_orderer`)** — hermes3:8b wins. 7.4x faster (15.9s vs 116s qwen3:14b), 0 hallucinated slugs, strict adherence to "minimal prereqs" rule in `_ORDERING_SYSTEM`. qwen3:14b over-included transitive prereqs (added `probability-and-distributions` to almost every chapter); hermes returned only the direct prereq, which is what the prompt actually asks for. Wired as `MODEL_ORDERER = MODEL_AGENT` in config.py. Verified 2026-05-14 on 8 ML chapters against curated ground truth.
- **Wiki clusterer (`wiki_builder._discover`)** — kept on qwen3:14b but **dropped `/no_think` suffix** as a bug fix. qwen3:14b in /no_think mode hallucinates readable-looking filenames ("GPT-3_paper.pdf") instead of carrying the actual input filenames through; downstream `(sources_dir / fn).exists()` check filters those out and clusters silently become empty. With /think (default), qwen3:14b is correct. hermes3:8b had real filenames but dropped 2/8 sources, so not yet a clusterer winner.
- **Wiki drafter (`wiki_builder.make_drafter`)** — kept on qwen3:14b. A/B on a 2-paper new-chapter draft: qwen3 produced a usable 3411-char draft with 8 wikilinks. hermes3:8b output was malformed (no `# heading`, fragmented prose, frontmatter `prerequisites: [\nself-slug\n]`, only 1 wikilink). gemma4:e4b produced 3 chars total — same JSON-string-with-newlines failure as prefill before we switched to `===SECTION===` markers. Drafter output is `SynthesisResult.markdown` (pydantic_ai structured output), so gemma is structurally unfit until the schema is reworked.
- **Briefing heavy tier (`briefing.call_heavy_tier`)** — kept on qwen3.5:35b via llama.cpp. A/B on real 4266-char context: qwen3.5:35b (186s) named specific companies + counts; qwen3:14b (50s) was vaguer with duplicates and leaked internal paths; hermes3:8b (20s) miscategorized LPL Financial as a job alert. Daily artifact + read-once-trusted = accuracy >> latency.
- **Prefill v2 Q&A path (`career/prefill.py`)** — gemma4:e4b confirmed winner 2026-05-15 on Twilio Segment SWE-L2 JD with 4 synthesized questions. gemma 50s, hermes 56s, qwen3:14b 122s (2.4x). Only gemma cited specific CV proof-points by metric ("Gaviali 1000+ req/s") in the why-company answer; hermes was generic; qwen invented stakeholder roles (PMs/designers) the candidate's profile doesn't have. Critical miss: qwen3:14b deferred the work-authorization question to `[NEEDS USER INPUT]` despite citizenship being documented in Personal Facts — over-conservatism, not safety. Both gemma and hermes correctly read the profile. Locks `MODEL_PREFILL = gemma4:e4b` for both cover-letter and Q&A paths.
- **Code review (`jef.py:cmd_review`)** — qwen3:14b beats hermes3:8b 2026-05-15 on a real `git diff` of jef.py changes. hermes3:8b hallucinated false positives twice in a row: (1) flagged `cmd_commit` for not validating staged-diff emptiness when the diff clearly shows the check; (2) flagged the `MODEL_AGENT` import line as a consistency issue when the diff was the import update itself. qwen3:14b correctly returned "No issues found" on a clean diff and caught a high-severity off-by-one bug on a deliberately buggy synthetic diff (`items[len(items)]`). Wired as `MODEL_WIKI_SYNTH` for `cmd_review` specifically — hermes is too noisy for code review; conservative > eager. Adds a new pattern: structured-output ≠ reasoning; hermes wins for grading because the output is a verdict tag, not a chain-of-reasoning over code.
- **Wiki verifier (`wiki_builder.verify_or_quarantine`)** — gemma4:e4b stays; qwen3:14b lost 2026-05-18. Motivation: gemma4:e4b crashes at retries=3 intermittently on the `QualityVerdict` structured output (already handled by a soft-pass that writes the page through with a warning). Tried `MODEL_VERIFIER = qwen3:14b` to remove the flake. On a fresh draft of the singing mixed-voice chapter, qwen3:14b (a) hit 180s timeout (bumped to 420s), then (b) returned a _wrong_ verdict (`faithful=False`, score=1) with issue text "Response is an article instead of a quality review" — it misread the rubric and graded the chapter for not being a review. Hard-quarantined a perfectly fine chapter. Reverted to `MODEL_VERIFIER = MODEL_PRIMARY`. The trade was strictly worse: gemma flake → soft-pass → page lands; qwen3 false negative → hard quarantine, no recourse. Inline note in `config.py:MODEL_VERIFIER` records the A/B so future sessions don't blindly retry the same swap. Real fix when this becomes worth doing: rewrite the verifier prompt so qwen3 understands the rubric, OR drop the verifier entirely for hybrid-merge output (already skipped) and only verify pure drafter output where verification has real value.

**Pattern:** Hermes excels at constrained-output decision tasks (grading, orderer) where "minimal" or "cautious" is the right behavior. Loses on open-ended prose generation (cover letters, drafter). Mixed on multi-item enumeration where coverage matters (clusterer). Gemma4:e4b is unfit for any task whose output is multi-paragraph text wrapped in a JSON string — use markdown section markers (`===SECTION===`) instead, see [[prefill]].

**How to apply:**

- Build the pipeline with a `--model` flag (or env override).
- Run the same input through at least 2 candidates: the obvious agentic pick (hermes3:8b) and the cheapest currently-loaded model (gemma4:e4b is usually it). For synthesis tasks, also try qwen3:14b.
- Compare: output validity, rule adherence (e.g. did it follow "verbatim" instruction?), sample rationale quality, latency per call.
- Don't default to Hermes just because it's "the agent model." Prose generation in particular favors gemma.
- If quality is a wash, default to whichever model is already loaded by adjacent pipelines to minimize swap thrash under `OLLAMA_MAX_LOADED_MODELS=1`.
- Note the format choice: gemma4:e4b is unreliable with `format: "json"` when a JSON string value needs embedded newlines (loses track of escaping). For multi-paragraph outputs, use markdown section markers like `===COVER_LETTER===` instead — works for both Hermes and gemma.
